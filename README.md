# AI 产品经理求职市场深度分析

> 用数据替代直觉，让求职决策有据可依。

基于北京地区 **943 条 AI/大模型产品经理岗位 JD** 的结构化深度分析。不是教你"怎么写简历"，而是回答：**该不该转 AI、该投哪个方向、该补什么能力、拿到 JD 怎么判断值不值得投。**

**[📊 点击查看 AI / 大模型产品经理岗位深度分析报告详情](https://aiisnothing.github.io/ai-pm-analysis/)**

> 报告为自包含 HTML（内嵌 ECharts），也可下载 [`report/AI产品经理_深度分析.html`](report/AI产品经理_深度分析.html) 到本地双击打开，无需联网。

## 为什么做这个项目

2026 年，"AI 产品经理"是招聘市场上最热门的方向之一。但热门不等于高薪，高薪不等于适合你。

作为一个正在考虑转型 AI 方向的产品经理，我面对的是一堆模糊的信号：铺天盖地的"AI 产品经理年薪百万"文章、招聘平台上参差不齐的 JD、社群里各执一词的过来人经验。**没有人基于大样本数据系统地回答过：这个市场到底长什么样？**

所以我决定自己动手——采集数据、清洗分析、写成可交互的报告。这个项目就是整个过程的沉淀。

## 我们做了什么

### 第一步：数据采集

从主流招聘平台采集了北京地区产品经理相关岗位的公开信息，覆盖 13 个产品方向（AI/大模型、策略、电商、增长、通用、企业服务等），总计 **7,201 条记录，4,780 条含完整 JD 正文**。

> ⚠️ 本仓库不包含爬虫代码和原始数据。采集部分涉及平台合规问题，仅分享分析方法和脱敏后的结论。

### 第二步：数据清洗与样本定义

原始数据噪声很大——薪资格式不统一（"25-40K·14薪"、"面议"）、方向分类粗糙、JD 质量参差。清洗规则：

```
薪资标准化：正则解析 "25-40K·14薪" → salary_min=25, salary_max=40, salary_mid=32.5
极端值剔除：salary_mid ≥ 100K 不参与统计（多为猎头虚高价）
样本筛选：3-10年经验 · 本科及以上 · salary_max ≥ 30K · 有完整 JD 正文
```

最终定义了两套分析口径：

| 口径 | 范围 | 样本量 | 用途 |
|------|------|--------|------|
| 🟢 主分析 | AI/大模型方向 | 943 条 | 深度分析 AI 岗位画像、能力要求、薪资因素 |
| 🟡 对照分析 | 全部 13 个方向 | 2,759 条 | AI vs 非 AI 横向对比 |

### 第三步：JD 结构化 — 从非结构化文本中提取信号

JD 是非结构化文本，无法直接做统计分析。我们用**正则规则**把每条 JD 转化为结构化特征向量：

**AI 技术信号**（16 类）：

```python
"大模型/LLM":   [r"大模型", r"llm", r"大语言模型", r"foundation\s*model"]
"Agent/智能体":  [r"agent", r"智能体", r"ai\s*agent"]
"RAG/检索增强":  [r"rag", r"检索增强", r"retrieval", r"知识库", r"向量"]
"Prompt工程":    [r"prompt", r"提示词", r"指令工程"]
"多模态":        [r"多模态", r"multi.?modal", r"视觉.*语言"]
# ... 共 16 类，详见 analysis/ai_deep_analysis.py
```

**产品能力信号**（12 类）：数据分析/SQL、项目管理/跨团队、场景抽象/系统设计、PRD/需求文档、用户研究、商业化/变现 ...

**工作内容信号**（10 类）：模型评测/效果优化、数据标注/语料管理、Prompt 设计/对话优化、算法策略/模型协同 ...

**软性要求**（8 类）：团队管理、从 0 到 1 经验、名校/高学历、AI 产品经验、技术/CS 背景 ...

每条 JD 最终变成一个 50+ 维的布尔/数值向量，可以做统计了。

### 第四步：岗位聚类 — 同样叫"AI 产品经理"，其实是 11 种不同的工作

"AI 产品经理"是一个笼统的标签。我们发现，同一个 title 下其实藏着非常不同的岗位。用**瀑布优先级规则**将 943 条 JD 归入 11 个岗位群：

```
优先级从高到低：
多模态 → 数字人/虚拟人 → AIGC/内容生成 → Agent/智能体 → 大模型平台 →
推荐/策略 → 数据标注/训练 → 对话/Chatbot → 搜索 → 行业AI → AI产品(通用)
```

为什么用规则而不是 ML 聚类？因为产品方向有明确的业务语义，规则可解释、可审计、可人工修正。ML 聚出来的 cluster 往往语义模糊，不利于给求职者建议。

**代价**：72% 的 JD 匹配了 2 个以上岗位群（比如一个 JD 同时提到 Agent 和 AIGC），优先级规则强制归入一个。我们为每个群标注了"分类稳定性"（高/中/低），让读者知道哪些群的边界比较模糊。

### 第五步：多维度交叉分析

有了结构化数据和聚类，开始做分析。核心指标体系：

| 指标 | 计算方式 | 回答什么问题 |
|------|----------|-------------|
| **P50/P75/P90** | 薪资中位数和分位数 | 这个方向/能力薪资水平如何？ |
| **高薪率** | salary_mid ≥ 45K 的 JD 占比 | 高薪机会有多密集？ |
| **信号出现率** | 含某关键词的 JD 数 ÷ 总 JD 数 | 市场对这个能力的需求有多普遍？ |
| **Lift** | 高薪组出现率 ÷ 普通组出现率 | 这个信号在高薪 JD 中是否更常见？ |
| **薪资溢价** | (含信号均薪 - 不含均薪) ÷ 不含均薪 | 有这个信号的 JD 平均贵多少？ |

但单一维度的分析容易被混杂因素误导。比如"多模态经验"和高薪正相关，到底是因为多模态本身值钱，还是因为要求多模态的岗位恰好都在大厂？

所以我们做了**分层控制变量观察**：

```
分层维度 × 信号 → 是否仍然正相关？
  ├── 3-5年 × 多模态 → lift = ?
  ├── 5-10年 × 多模态 → lift = ?
  ├── 大厂 × 多模态 → lift = ?
  └── 非大厂 × 多模态 → lift = ?

如果 4 个分层中 3+ 个方向一致 → 该信号"稳健"
如果方向不一致 → 该信号可能被混杂因素驱动
```

这不是严格的因果推断（做不到，观察数据没有随机分配），但比不控制变量直接下结论靠谱得多。

### 第六步：从分析到可执行建议

分析结果如果只是一堆数字，求职者拿到了也不知道该干嘛。所以每个结论都改成了**判断 + 依据 + 行动**的三段结构：

```
判断：AI 方向的高薪率显著高于大盘，是当前薪资竞争力最强的产品方向。
依据：AI P50 = 40K，高薪率 44.1%，高于大盘基准（29.2%）14.9pp，排名第一。
行动：如果你有技术理解力和 AI 动手能力，AI 方向值得优先投入。但方向内部差异很大，需要选对细分方向。
```

还做了几个直接落地的工具：
- **JD 评分卡**：拿到一条 JD，4 个维度打分判断值不值得投
- **2×2 判断矩阵**：岗位质量 × 个人匹配度，四象限决策
- **作品集交付模板**：14 步从选题到交付的完整流程
- **背景适配表**：5 类背景 × 适合什么方向 × 可迁移能力 × 最大短板

### 第七步：报告可视化

用 ECharts 做了 22 张交互图表，打包成一个自包含 HTML 文件（内嵌 JS，离线可用）。暗色主题，每张图表旁边都有指标解释和解读。

## 走过的弯路和方法论反思

做这个项目最大的收获不是结论本身，而是过程中踩的坑：

### 1. 出现率高 ≠ 重要，出现率低 ≠ 不重要

PRD/需求文档在高薪 JD 中出现率（28%）低于普通 JD（43%）。早期版本据此说"PRD 是低价值能力"。

但这是错的——高薪岗默认你已经会写 PRD，所以不在 JD 里强调。不提不代表不需要。

**教训：JD 是信号，不是全集。"没写"可能是"默认你会"，也可能是"真的不需要"，需要结合语境判断。**

### 2. 堆叠柱状图的总高度要有语义

早期版本把"硬性要求出现率"和"加分项出现率"做成堆叠柱状图。问题是：硬性 + 加分 = 总出现率，这个合计值不是求职者最关心的指标，堆在一起反而让人误以为柱子越长越重要。

改成了柱状图（硬性要求）+ 折线图（加分占比）双轴，两个维度各自清晰。

**教训：可视化不只是"展示数据"，图表形式本身在传达判断。选错图表类型 = 给读者植入错误直觉。**

### 3. 每个结论都要经得起"换个角度问"

报告经历了 6 个大版本的迭代（V19→V24），每一轮都是被自己或用户的追问打回去重写。最典型的追问是：

- "你说 AI 和策略差不多——为什么只跟策略比？"
- "你说 Prompt 不重要——那为什么 83% 的 JD 写了它是硬性要求？"
- "你说大厂高薪率高——是大厂让人值钱，还是大厂本来就招贵的人？"

**教训：如果你的结论经不起一个简单的反问，它就不该出现在报告里。**

## 主要发现

| # | 发现 | 数据支撑 |
|---|------|----------|
| 1 | **AI 方向高薪率显著高于大盘** | AI P50 = 40K，高薪率 44.1%，高于大盘基准（29.2%）14.9pp，在所有方向中排名第一 |
| 2 | **方向选择可能比技能更重要** | 11 个岗位群高薪率从 26% 到 57%，差距 31pp，远大于任何单一技能 |
| 3 | **Prompt/RAG 是门槛不是卖点** | 83%+ JD 列为硬性要求，但高薪组出现率反而低于普通组（lift 0.50-0.85） |
| 4 | **经验和平台的影响大于技能** | 5-10年高薪率 51% vs 3-5年 24%；万人大厂 57% vs 小公司 16% |
| 5 | **高薪 JD 聊"你能负责什么"** | PRD/原型在高薪 JD 中出现率更低（0.52-0.64x），团队管理更高（1.75x） |

## 如果你想做类似的分析

这套方法不限于"AI 产品经理"，任何方向的招聘数据分析都可以复用：

1. **定义信号字典**：根据你的目标方向，定义要从 JD 中提取哪些关键词（参考 `ai_deep_analysis.py` 中的 `AI_TECH_SIGNALS` 和 `PRODUCT_CAP_SIGNALS`）
2. **定义聚类规则**：如果你的方向内部有细分（比如前端里的 React/Vue/全栈），写瀑布优先级规则
3. **定义高薪阈值**：用 P75 或你认为合适的分位数作为高薪线
4. **跑信号出现率 + Lift + 分层观察**：核心分析三件套
5. **对每个结论做反问测试**：经不起反问的结论不要写进报告
